# PQSend v0.1.0-alpha.1

`v0.1.0-alpha.1` is the first experimental encrypted package alpha for PQSend.
It provides a narrow local CLI workflow for encrypting one file to one explicit
age X25519 recipient and opening it with one explicit X25519 identity.

> [!WARNING]
> This alpha is experimental, unaudited, X25519-only, not
> post-quantum-secure, and not ready for sensitive real-world data. The
> `.pqsend` package format is unstable and may change incompatibly before
> `v1.0.0`.

## Included

- age-backed binary age v1 adapter for exactly one X25519 recipient
- fixed 20-byte `.pqsend` public envelope
- one-file `.pqsend` package creation and authenticated opening
- encrypted internal single-file manifest
- original filename hidden from public package metadata
- explicit recipient and identity key-file CLI workflow
- `keygen`, `pack`, `open`, and `inspect` commands
- local human-readable security receipts
- tests for tampering, truncation, wrong identity, filename leakage, inspect
  leakage, unsafe filenames, and overwrite refusal

The public `inspect` command validates and displays only the public envelope and
total package size. It does not decrypt the package or reveal the encrypted
original filename.

## Security and privacy notes

- PQSend delegates recipient encryption and authenticated payload protection to
  the Rust `age` crate. It does not invent cryptographic primitives.
- The v0.1 file-size limit is 64 MiB (`67,108,864` bytes).
- The original filename is encrypted inside the package. The outer `.pqsend`
  filename is public transport metadata, so users who want to avoid filename
  leakage must not name the outer package after the original file.
- The public envelope reveals the format version, single-file mode, age/X25519
  backend, exact encrypted payload length, and approximate plaintext size.
- Package creation, key generation, and extraction refuse to overwrite
  existing files.
- Opening validates the complete authenticated package and rejects unsafe
  filenames before publishing plaintext output.
- Local security receipts are not signatures, proof of authorship, proof of
  delivery, or external proof.

## Not included

- post-quantum support
- contact integration with package operations
- folder support
- multiple recipients
- sender signatures
- password mode
- GUI
- relay server
- Wi-Fi transfer
- chat
- a stable package format
- an external security audit

## Known limitations

- X25519 does not protect against harvest-now-decrypt-later or other
  post-quantum adversaries.
- The implementation and package format have not received an external security
  audit.
- Pre-`v1.0.0` packages may become incompatible with later releases.
- Files, plaintext, encrypted payloads, and packages are buffered in memory at
  different stages; temporary and in-memory plaintext remnants cannot be
  guaranteed erased.
- Approximate plaintext size, package size, transfer timing, and outer package
  filename may leak.
- PQSend does not protect compromised endpoints, substituted or poorly
  protected keys, recipient plaintext disclosure, or all denial-of-service
  attacks.
